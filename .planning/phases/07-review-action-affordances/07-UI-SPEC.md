## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps, desktop review-action button gaps, title/help adjacency |
| sm | 8px | Button padding, mobile action gaps, prompt/recovery status rhythm |
| md | 16px | Default mobile shell and row spacing inherited from Phase 06 |
| lg | 24px | Section spacing only when an existing surface needs separation |
| xl | 32px | Wide layout gaps only; do not add in Phase 07 |
| 2xl | 48px | Reserved for page-level breaks; do not add in Phase 07 |
| 3xl | 64px | Reserved for page-level breaks; do not add in Phase 07 |

Exceptions: mobile review-action buttons keep the Phase 06 `44px` minimum width and height. Desktop
review buttons may grow to fit labels but must keep a stable minimum height of `32px`, `4px 8px`
padding, and `4px` button gap. No other non-grid spacing is allowed.

## Visual Hierarchy

Phase 07 changes the affordance of the review actions only. The Compliance surface remains the
primary review surface, the selected review row remains the focus anchor, and the existing Phase 06
stacked mobile row layout remains unchanged.

### WUI Action Presentation

| Element | Contract |
|---------|----------|
| Action order | Approve, Reject, Changes. Keep this order in buttons, hotkey help, and tests. |
| Visible labels | Use `A Approve`, `X Reject`, and `C Changes` on actionable WUI buttons. Do not use bare `A`, `X`, `C` after this phase. |
| Title text | `Approve review (a)`, `Reject review - comment required (x)`, `Request changes - details required (c)`. |
| Accessible names | `Approve review for {task_id}`, `Reject review for {task_id}`, `Request changes for {task_id}`. |
| Desktop sizing | Buttons are text buttons with `min-height: 32px`, `padding: 4px 8px`, `border-radius: 2px`, and no icon-only treatment. |
| Mobile sizing | Keep Phase 06 `min-width: 44px`, `min-height: 44px`, `padding: 8px`, wrapping action row, and full-width `Actions` cell. |
| Read-only rows | Keep the existing `read-only` copy. Do not render disabled fake action buttons. |

Button color remains semantic and restrained: Approve uses the ok border, Reject uses the destructive
border, and Changes uses the warning border. Do not fill all buttons with color by default.

### TUI Action Presentation

| Element | Contract |
|---------|----------|
| Action cell | Actionable review rows may keep compact `a/x/c` text in the `Actions` column. |
| Footer help | The TUI caption must spell out `a=Approve review  x=Reject review  c=Changes` so operators can understand the hotkeys without external docs. |
| Selection marker | Keep the `>` marker for the selected actionable review gap. |
| Expert speed | Keep single-key `a`, `x`, and `c` actions. Do not add blocking text entry or multi-step modal behavior to the TUI in this phase. |
| Feedback | Error/status copy must name the action or missing prerequisite when a review action cannot be recorded. |

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 13px | 400 | 1.45 |
| Label | 12px | 600 | 1.2 |
| Heading | 15px | 600 | 1.2 |
| Display | 18px | 600 | 1.2 |

Use only weights `400` and `600` for newly touched WUI labels, action help, and recovery copy.
Do not scale font size with viewport width. Long task IDs, stage IDs, reviewer emails, and prompt
feedback must wrap with `overflow-wrap: anywhere` where they appear in the WUI.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Light `#f7f8fa`; dark `#111315` | Page background and operator-console negative space |
| Secondary (30%) | Light `#ffffff` / `#f1f3f6`; dark `#171a1d` / `#202428` | Tables, stacked mobile rows, metric strip, tabs, controls |
| Accent (10%) | `#0f766e` | Selected review row outline, keyboard/touch focus outline, subtle row hover/active background |
| Destructive | `#b42318` | Reject action border/status, failed status, rejected review state, destructive recovery copy |

Accent reserved for: selected actionable review row outline, keyboard/touch focus outline, and subtle
row hover/active background. Do not use accent for all review buttons, table borders, or generic
labels.

Existing semantic status colors remain data-state colors only: ok `#16703c`, warning `#a15c07`,
info `#0969da`, destructive `#b42318`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Approve review` |
| Empty state heading | none - empty states remain a single row inside the affected table |
| Empty state body | Existing copy: `No human review or verification gaps.` |
| Error state | `Review not recorded: {detail}. Check admin token, reviewer, and selected row, then retry.` |
| Destructive confirmation | `Reject review`: prompt `Reject review: enter required comment`; blank feedback `Reject requires a comment`; cancel feedback `Reject canceled` |
| Blocking confirmation | `Changes`: prompt `Request changes: describe required changes`; blank feedback `Changes requires details`; cancel feedback `Changes canceled` |

Additional required status copy:

| Situation | Copy |
|-----------|------|
| No selected WUI review row | `No review row selected` |
| Missing admin token | `Admin token required` |
| Missing reviewer | `Reviewer required` |
| WUI approve success | `Review recorded: approved` |
| WUI reject success | `Review recorded: rejected` |
| WUI changes success | `Review recorded: changes requested` |
| TUI missing reviewer | `reviewer required: pass --reviewer or set WHILLY_OPERATOR_EMAIL` |
| TUI no selected row | `no actionable human review gap selected` |

Do not add true undo copy in Phase 07. Undo remains deferred because it needs backend/audit reversal
semantics.

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| WUI click actions | Button clicks continue reading `data-review-decision` and the enclosing `tr[data-review-task]`. |
| WUI keyboard actions | `a`, `x`, and `c` continue to submit only on the Compliance surface and only outside editable inputs. |
| Decision literals | Keep `approved`, `rejected`, and `changes_requested` exactly. |
| API path | Keep `POST /api/v1/tasks/{task_id}/human-review` exactly. |
| Shared service path | Preserve the shared `record_review_decision` pipeline-layer command path. |
| Reject prompt | WUI `rejected` must require a non-empty trimmed comment before `fetch` is called. Cancel or blank input leaves the task unchanged. |
| Changes prompt | WUI `changes_requested` must require one non-empty trimmed requested-changes entry before `fetch` is called. Cancel or blank input leaves the task unchanged. |
| Approve path | WUI `approved` must not prompt for a comment and must keep expert hotkey speed. |
| Recovery feedback | Cancel, blank prompt, missing token, missing reviewer, no row, HTTP error, and network error must update the existing polling/status region rather than silently doing nothing. |
| Refresh | Successful decisions still trigger the existing `whilly:reviewDecision` refresh path and preserve active surface, filter text, selected review row, and focus behavior. |
| Mobile behavior | Below `900px`, action labels may wrap but must not overlap row text or shrink below the `44px` touch target. |
| TUI hotkeys | Preserve `handle_tui_key` mappings for `a`, `x`, and `c`; keep decisions single-key and Compliance-only. |
| TUI request changes | The existing browserless fixed requested-changes note is acceptable: `Requested from TUI operator controls.` |

## Implementation Boundaries

| Item | Contract |
|------|----------|
| Backend review endpoint | Do not change the route, auth model, response contract, or event type mapping. |
| Audit payload shape | Do not add new fields beyond already-supported `comment` and `requested_changes`. |
| WUI storage | Do not add persistent browser storage for review recovery. Existing dashboard session state remains unchanged. |
| Dialog model | Use browser-native prompts only. Do not add modal frameworks, custom dialog components, or new JS libraries. |
| Table contract | Preserve Phase 05/06 table labels, field order, row ids, table ids, `data-mobile-label`, `data-filter-text`, and mobile CSS scoping. |
| Pause/resume | Do not change pause/resume controls, labels, API paths, or worker-control semantics. |
| TUI layout | Keep compact Rich tables. Do not replace the TUI with a new layout framework. |

## Testing Contract

Phase 07 tests must pin the UI contract without asserting unrelated rendering details.

| Surface | Required assertions |
|---------|---------------------|
| WUI template | Visible labels `A Approve`, `X Reject`, `C Changes`; title text; aria labels; decision literals; existing endpoint path. |
| WUI JavaScript | Reject and Changes cancel/blank paths call `setPollingStatus` and return before `fetch`; success status names the recorded decision. |
| WUI hotkeys | `a/x/c` still map to the same decision literals on the Compliance surface only. |
| WUI mobile CSS | Existing 900px stacked-row rules and 44px action-button minimums remain present. |
| TUI render | Caption spells out `a=Approve review`, `x=Reject review`, and `c=Changes`; action cell remains compact. |
| TUI key handling | `a`, `x`, and `c` still set `approved`, `rejected`, and `changes_requested`; no new blocking input state is required. |
| Shared path | TUI review recording still calls the shared service with the existing source, stage, reviewer, and decision fields. |
